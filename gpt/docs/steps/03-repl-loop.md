# 03. REPL 루프 추가

## 이번 단계의 목표

기존 CLI는 두 가지 방식으로 동작했다.

- 인자를 주면 한 번 실행하고 끝나는 단일 프롬프트 모드
- 스트리밍 응답 출력

이번 단계의 목표는 여기에 **대화형 REPL 루프**를 추가해서, 터미널 안에서 계속 입력하고 계속 응답받는 흐름을 만드는 것이다.

즉 이번 단계의 핵심은 다음 한 줄로 정리할 수 있다.

**단발성 CLI를, 터미널 안에서 반복적으로 대화할 수 있는 CLI로 확장한다.**

## 왜 이 단계가 필요한가

하네스는 단순히 한 번 호출하고 끝나는 실행기로만 머물지 않는다.  
실제 작업형 하네스는 사용자가 계속 요청을 입력하고, 그에 따라 여러 번 모델을 호출하며, 점점 더 복잡한 상호작용을 만들어낸다.

REPL은 그 출발점이다.

이번 단계가 중요한 이유:

- 사용자가 매번 명령을 다시 실행하지 않아도 된다.
- 하네스가 “대화형 도구”처럼 보이기 시작한다.
- 이후 대화 이력, 도구 호출, 상태 유지 같은 기능을 붙일 기반이 된다.
- 사용자 입력 루프와 모델 호출 루프를 분리해서 생각할 수 있게 된다.

즉 이번 단계는 하네스를 단순한 실행 스크립트에서 `대화형 CLI`로 한 단계 올리는 작업이다.

## 이번 단계의 범위

### 포함

- 인자 없이 실행하면 REPL 모드로 진입
- 한 줄 입력마다 Anthropic 스트리밍 호출 실행
- `exit` 또는 `quit` 입력 시 종료
- 빈 입력은 무시하고 다음 프롬프트 표시
- REPL 동작을 검증하는 자동 테스트 추가

### 제외

- 대화 이력 유지
- 이전 질문/응답을 다음 호출에 전달하는 기능
- 도구 호출
- 세션 저장
- 멀티 에이전트

## 현재 구조와 이번 단계의 변화

기존 구조:

```text
단일 프롬프트 입력
-> 설정 로드
-> 스트리밍 호출
-> 응답 출력
-> 종료
```

변경 후 구조:

```text
프로그램 시작
-> 인자 유무 확인
-> 인자가 있으면 단일 프롬프트 모드
-> 인자가 없으면 REPL 모드
-> 한 줄 입력
-> 스트리밍 호출
-> 응답 출력
-> 다음 입력 대기
-> exit/quit 시 종료
```

핵심 차이는 하네스가 “한 번 실행되고 끝나는 구조”에서 “사용자가 종료할 때까지 계속 입력을 받는 구조”로 바뀌었다는 점이다.

## 실제로 바뀐 파일

- `src/cli.ts`
- `src/lib/run-repl-command.ts`
- `tests/repl-cli.test.ts`
- `README.md`

## 핵심 구현 내용

### 1. CLI 라우팅 분기

`src/cli.ts`에서 프로그램 시작 시 인자 유무를 먼저 확인한다.

- 인자가 있으면 기존 `runPromptCommand()` 사용
- 인자가 없으면 새로 추가한 `runReplCommand()` 사용

즉 이제 하나의 CLI가 두 가지 모드를 가진다.

- 단일 프롬프트 모드
- REPL 모드

### 2. REPL 실행 함수 추가

`src/lib/run-repl-command.ts`를 새로 추가했다.

이 파일은 다음 책임을 맡는다.

- REPL 시작 메시지 출력
- 설정 로드
- 프롬프트(`> `) 표시
- 줄 단위 입력 처리
- 빈 입력 무시
- `exit`, `quit` 처리
- 각 줄마다 스트리밍 호출 실행
- 오류 발생 시 stderr 출력 및 종료 코드 반환

즉 REPL의 핵심 제어 흐름은 이 파일에 모여 있다.

### 3. readline 기반 입력 처리

실제 터미널 입력은 `src/cli.ts`에서 Node의 `readline` 인터페이스를 열어 처리한다.

이렇게 하면:

- 실제 터미널 환경에서는 표준 입력으로 계속 읽을 수 있고
- 테스트에서는 별도의 가짜 입력 스트림으로 `runReplCommand()`를 독립 검증할 수 있다

즉 “실제 입력 장치”와 “REPL 동작 로직”을 분리했다는 점이 중요하다.

### 4. 자동 테스트 추가

`tests/repl-cli.test.ts`에서 아래 동작을 검증한다.

- REPL이 각 줄을 받아 스트리밍 호출하는가
- 빈 입력을 무시하고 다음 프롬프트를 다시 보여주는가
- 스트리밍 오류 시 stderr를 출력하고 종료하는가

이 단계에서는 REPL의 핵심 제어 흐름을 테스트로 먼저 고정하고 구현했다.

## REPL 동작 규칙

이번 단계에서 정한 규칙은 다음과 같다.

- 인자 없이 실행하면 REPL 시작
- 프롬프트는 `> `
- 빈 입력은 무시
- `exit`, `quit` 입력 시 종료
- 각 줄은 독립된 단일 요청으로 처리
- 아직은 이전 대화를 기억하지 않음

즉 이번 REPL은 “대화형 모양”은 갖췄지만, 아직은 상태를 유지하지 않는 REPL이다.

## 실행 방법

### REPL 모드

```bash
npm run dev
```

예시:

```text
REPL 모드입니다. 종료하려면 exit 또는 quit를 입력하세요.
> 안녕하세요
...스트리밍 응답...
> quit
REPL을 종료합니다.
```

### 단일 프롬프트 모드

```bash
npm run dev -- "안녕하세요. 두 문장으로 짧게 자기소개를 해주세요."
```

### 테스트

```bash
npm test
```

아래 조건을 모두 만족하면 이번 단계는 완료다.

- 인자 없이 실행 시 REPL 모드가 시작된다.

## 메모

이번 단계의 REPL은 아직 상태를 유지하지 않는다.  
즉 사용자는 여러 번 입력할 수 있지만, 모델은 각 줄을 독립 요청으로 처리한다.

따라서 다음 단계의 핵심 질문은 이것이 된다.

**REPL은 만들어졌으니, 이제 이전 대화를 어떻게 기억하게 만들 것인가?**
