# 00. 하네스란 무엇인가

## 하네스의 정의

이 문맥에서 하네스(harness)는 단순히 하나의 모델이나 특정 제품 이름을 뜻하지 않는다.  
하네스는 **언어 모델이 실제 작업을 수행하도록 입력, 상태, 도구, 규칙, 검증을 연결하는 실행 계층**을 의미한다.

조금 더 쉽게 말하면 다음과 같다.

- 모델은 생각하고 응답을 만든다.
- 하네스는 그 모델이 어떤 방식으로 생각하고, 어떤 도구를 쓰고, 어떤 규칙을 따르며, 어떤 흐름으로 일할지를 정한다.

즉 하네스는 모델 자체가 아니라, **모델을 실제로 일하게 만드는 운영 구조**다.

## 왜 하네스가 필요한가

LLM은 기본적으로 매우 강력한 텍스트 생성기지만, 그 자체만으로는 실제 작업을 안정적으로 수행하기 어렵다.

예를 들면 모델 하나만 두었을 때는 아래와 같은 문제가 생긴다.

- 어떤 입력을 얼마나 넣어야 하는지 불분명하다.
- 긴 작업에서 문맥이 쉽게 섞이거나 오염된다.
- 파일, 셸, 웹 같은 외부 환경을 직접 다룰 수 없다.
- 잘못된 출력을 내도 자동으로 검증하지 못한다.
- 프로젝트마다 다른 규칙을 자연스럽게 따르기 어렵다.

하네스는 이런 문제를 해결하기 위해 존재한다.  
즉 하네스는 모델이 “그냥 대답하는 존재”에 머무르지 않고, **일정한 규칙 안에서 실제 작업을 수행하는 존재**가 되도록 만든다.

## 하네스의 핵심 구성요소

하네스는 보통 아래 요소들로 이루어진다.

### 1. 입력 계층

사용자 요청, 시스템 프롬프트, 프로젝트 규칙, 현재 작업 문맥을 모아 모델에 전달하는 부분이다.

예:

- 사용자 메시지
- 시스템 지침
- 로컬 규칙 파일
- 현재 열려 있는 작업 정보

### 2. 상태 계층

대화 이력, 세션 정보, 중간 결과 같은 작업 상태를 관리하는 부분이다.  
LLM은 기본적으로 상태를 스스로 오래 유지하지 못하기 때문에, 이 계층이 없다면 긴 작업에서 쉽게 맥락을 잃는다.

### 3. 모델 호출 계층

실제로 LLM API를 호출하고, 응답을 받아서 해석하는 부분이다.

예:

- 일반 텍스트 응답 받기
- 스트리밍 응답 처리
- 구조화된 출력 파싱
- tool call 감지

### 4. 도구 실행 계층

모델이 요청한 외부 도구를 실제로 실행하는 부분이다.

예:

- 파일 읽기/쓰기
- 셸 명령 실행
- 웹 검색
- Git 작업
- MCP 도구 호출

### 5. 정책 및 권한 계층

무엇을 허용하고 무엇을 제한할지 정하는 부분이다.

예:

- 위험한 명령 차단
- 샌드박스 적용
- timeout 적용
- 특정 폴더 접근 제한

### 6. 검증 계층

모델이 낸 결과가 실제로 맞는지 확인하는 부분이다.

예:

- 테스트 실행
- 형식 검증
- 결과 확인
- 실패 시 재시도

### 7. 오케스트레이션 계층

작업을 한 번에 던지지 않고, 여러 단계로 나누거나 여러 에이전트에 역할을 분배하는 부분이다.

예:

- 요구사항 정리
- 계획 수립
- 구현
- 테스트
- 검토

좋은 하네스일수록 이 흐름을 잘게 나누고, 각 단계에서 필요한 정보만 다루도록 만들어 컨텍스트 오염을 줄인다.

## 하네스는 어떻게 동작하는가

가장 단순한 하네스의 흐름은 다음과 같다.

```text
사용자 입력
-> 입력 정리
-> 모델 호출
-> 응답 출력
```

하지만 실제 작업형 하네스는 보통 아래처럼 더 많은 단계를 가진다.

```text
사용자 요청
-> 요청 해석
-> 필요한 문맥과 규칙 주입
-> 모델 호출
-> 도구 요청 감지
-> 도구 실행
-> 결과를 다시 모델에 전달
-> 출력 생성
-> 검증
-> 필요하면 수정 및 재시도
```

즉 하네스의 본질은 단순한 “API 호출기”가 아니다.  
하네스는 **모호한 인간의 요청을 실제 실행 가능한 작업 흐름으로 바꾸는 시스템**이다.

## 하네스와 자주 헷갈리는 개념

### 모델

모델은 텍스트를 생성하고 추론하는 엔진이다.  
모델은 하네스 안에서 동작하는 핵심 부품이지만, 하네스 그 자체는 아니다.

### 에이전트

에이전트는 목표를 가지고 여러 단계의 행동을 수행하는 실행 주체다.  
하네스는 이 에이전트가 어떻게 움직일지를 정하는 바깥 구조라고 볼 수 있다.

### 도구

도구는 특정 기능을 수행하는 실행 수단이다.

예:

- 파일 읽기
- 파일 쓰기
- 셸 실행
- 테스트 실행

도구는 하네스의 일부일 수는 있지만, 도구 하나만으로 하네스가 되지는 않는다.

### 제품

`Claude Code`, `Codex CLI` 같은 것은 실제 제품 또는 구현체다.  
이 제품들 안에는 각각의 하네스가 들어 있다.

즉 제품은 “완성된 시스템”이고, 하네스는 “그 시스템의 운영 구조”에 가깝다.

## 좋은 하네스의 특징

좋은 하네스는 단순히 모델 호출이 되는 것만으로 충분하지 않다.

좋은 하네스는 보통 아래 특징을 가진다.

- 입력을 일관되게 정리한다.
- 필요한 문맥만 넣어 컨텍스트 오염을 줄인다.
- 도구를 안정적으로 연결한다.
- 위험한 행동을 제한한다.
- 결과를 검증한다.
- 실패했을 때 다시 고칠 수 있는 루프가 있다.
- 프로젝트나 사용자에 맞게 행동을 조정할 수 있다.

결국 좋은 하네스는 **좋은 모델을 더 잘 쓰게 만들고, 나쁜 흐름 때문에 성능이 무너지지 않도록 지켜주는 구조**라고 볼 수 있다.

## 이 프로젝트에서의 의미

이 프로젝트는 거대한 하네스를 한 번에 만들기보다, 가장 작은 실행 루프부터 차근차근 구현하면서 하네스의 각 계층을 직접 이해하는 것을 목표로 한다.

즉 이 문서는 구현 계획서가 아니라, 이후 단계 문서들을 읽기 위한 기준 문서다.  
앞으로 추가될 문서들은 모두 “하네스의 어느 부분을 구현하는가”라는 관점에서 이해하면 된다.

## 요약

하네스는 다음 한 문장으로 정리할 수 있다.

**하네스는 언어 모델이 실제 작업을 수행하도록 입력, 상태, 도구, 규칙, 검증을 연결하는 실행 계층이다.**

그리고 더 쉽게 말하면 다음과 같다.

**하네스는 모델을 실제로 일하게 만드는 시스템이다.**
