# 05. 세션 별 저장 추가

## 이번 단계의 목표

이전 단계에서는 REPL 세션이 살아 있는 동안만 대화 이력을 기억했다.  
프로그램을 종료하면 메모리에만 있던 이력이 사라졌기 때문에, 같은 작업을 다음 실행에서 이어갈 수는 없었다.

이번 단계의 목표는 **세션 이름별로 대화 이력을 파일에 저장하고, 다음 실행에서 다시 불러오는 것**이다.

즉 이번 단계가 끝나면 다음 흐름이 가능해진다.

- `npm run dev`로 기본 세션을 열고 대화한다.
- REPL을 종료한다.
- 다시 `npm run dev`를 실행하면 이전 대화가 복원된다.

또한 선택적으로 세션 이름을 지정해 서로 다른 대화창처럼 분리할 수 있다.

- `npm run dev -- --session bugfix-chat`
- `npm run dev -- --session docs-work`

## 왜 이 단계가 필요한가

하네스에서 세션은 단순한 저장 기능이 아니라 **작업 흐름의 경계** 역할을 한다.

세션 별 저장이 필요한 이유:

- 작업을 중간에 멈췄다가 다시 이어갈 수 있다.
- 서로 다른 목적의 대화를 분리해 문맥 오염을 줄일 수 있다.
- 실제 채팅 앱처럼 “대화창을 다시 여는 경험”을 만들 수 있다.
- 이후 도구 호출, 권한, 검증 같은 기능을 세션 단위로 확장하기 쉬워진다.

이번 단계는 하네스가 메모리 안의 임시 상태를 넘어, **지속되는 작업 공간**을 갖기 시작하는 첫 단계다.

## 이번 단계의 범위

### 포함

- REPL 시작 시 세션 파일이 있으면 기존 대화 이력을 불러오기
- REPL에서 한 턴이 끝날 때마다 세션 파일에 이력 저장하기
- 기본 세션 이름을 `default`로 사용하기
- `--session <이름>` 플래그로 다른 세션을 선택할 수 있게 하기
- 세션 파일을 `data/sessions/<세션이름>.json` 형태로 저장하기
- 세션 저장/복원 동작을 검증하는 자동 테스트 추가하기

### 제외

- 세션 목록 조회 명령
- 세션 삭제/이름 변경
- 이력 요약이나 압축
- 단일 프롬프트 모드에서 세션 저장 사용
- 도구 호출

## 세션 파일 구조

이번 단계에서는 가장 단순한 JSON 구조를 사용한다.

```json
{
  "id": "default",
  "messages": [
    { "role": "user", "content": "안녕" },
    { "role": "assistant", "content": "안녕하세요." }
  ],
  "updatedAt": "2026-04-08T14:00:00.000Z"
}
```

핵심은 `messages` 배열이다.  
현재 REPL이 메모리에 들고 있던 대화 이력을 거의 그대로 파일에 옮겨놓는 구조다.

## 현재 구조에서 바뀌는 점

기존 REPL 구조:

```text
REPL 시작
-> 메모리 history = []
-> 사용자 입력
-> 모델 호출
-> history 갱신
-> 종료 시 history 소멸
```

변경 후 REPL 구조:

```text
REPL 시작
-> 세션 이름 결정
-> 세션 파일 읽기
-> history 복원
-> 사용자 입력
-> 모델 호출
-> history 갱신
-> 세션 파일 저장
-> 종료 후에도 history 유지
```

## 구현 방향

### 1. 세션 저장소 모듈 분리

파일 입출력 책임은 `src/lib/session-store.ts`로 분리한다.

이 모듈은 다음 역할을 맡는다.

- 세션 이름을 받아 저장 경로 계산
- 세션 파일이 없으면 빈 이력 반환
- 세션 파일이 있으면 메시지 배열 복원
- 이력을 JSON 파일로 저장

이렇게 분리하면 REPL 로직은 “언제 저장하고 언제 불러올지”만 담당하고, 파일 형식과 경로 계산은 저장소 모듈에 맡길 수 있다.

### 2. REPL은 세션 저장소를 의존성으로 받기

`runReplCommand()`는 직접 파일을 읽고 쓰지 않고, 다음 같은 의존성을 받도록 확장한다.

- 세션 이력 불러오기
- 세션 이력 저장하기

이 구조를 쓰면 테스트에서 파일 시스템 없이도 REPL 동작을 검증할 수 있다.

### 3. CLI에서 세션 이름 해석

기본 실행:

```bash
npm run dev
```

이 경우 `default` 세션을 연다.

세션 지정 실행:

```bash
npm run dev -- --session bugfix-chat
```

이 경우 `bugfix-chat` 세션 파일을 읽고 저장한다.

이번 단계에서는 `--session` 플래그는 REPL 모드에서만 사용한다.

## 검증 계획

이번 단계가 끝나면 아래를 확인한다.

### 자동 테스트

- 저장된 세션 이력이 다음 실행에서 복원되는지
- REPL이 복원된 이력을 첫 모델 호출에 포함하는지
- 한 턴이 끝날 때마다 세션 파일 저장 함수가 호출되는지
- 세션 파일이 없을 때 빈 이력으로 시작하는지

### 수동 테스트

1. `npm run dev`
2. 질문 1개 입력 후 종료
3. 다시 `npm run dev`
4. 이전 질문을 기억하는지 확인

그리고 세션 분리도 확인한다.

1. `npm run dev -- --session alpha`
2. `npm run dev -- --session beta`
3. 서로 다른 대화를 했을 때 문맥이 섞이지 않는지 확인

## 실제 구현 내용

이번 단계에서 실제로 반영한 내용은 아래와 같다.

### 1. CLI 인자 파서 추가

`src/lib/cli-options.ts`를 추가해서 CLI 실행 방식을 명시적으로 해석하도록 만들었다.

- 인자가 없으면 `default` 세션의 REPL 모드
- `--session <이름>`이면 해당 세션의 REPL 모드
- 일반 인자가 있으면 단일 프롬프트 모드

이번 단계에서는 `--session` 플래그와 단일 프롬프트 인자를 함께 쓰는 경우는 허용하지 않는다.

### 2. 파일 기반 세션 저장소 추가

`src/lib/session-store.ts`를 추가했다.

이 모듈이 맡는 역할:

- `data/sessions/<세션이름>.json` 경로 계산
- 세션 파일이 없으면 빈 이력 반환
- 세션 파일이 있으면 기존 메시지 배열 복원
- 한 턴이 끝날 때마다 JSON 파일로 저장
- 위험한 세션 이름 거부

즉 이번 단계부터 REPL은 메모리뿐 아니라 파일 시스템에도 대화 이력을 남긴다.

### 3. REPL에 세션 복원/저장 연결

`src/lib/run-repl-command.ts`는 이제 세션 이름을 옵션으로 받고, 시작할 때 세션 이력을 불러온다.

동작 흐름:

1. REPL 시작
2. 세션 이력 복원
3. 사용자 입력 처리
4. 모델 응답 스트리밍
5. user/assistant 턴을 이력에 추가
6. 세션 파일 저장

복원된 이력이 있으면 몇 개의 메시지를 복원했는지 안내 문구도 출력한다.

### 4. README와 사용법 갱신

README에 아래 내용을 반영했다.

- 현재 단계가 `05. 세션 별 저장 추가`라는 점
- `npm run dev -- --session <이름>` 사용법
- 세션 파일이 `data/sessions/` 아래에 저장된다는 점
