# 사내 채팅 시스템 요구사항 명세서

## 1. 프로젝트 개요

본 시스템은 기업 내부 구성원 간의 원활한 커뮤니케이션을 지원하기 위한 **사내 채팅 플랫폼**이다.
사용자는 1:1 대화, 그룹 채팅, 부서별 채널, 공지 채널 등을 통해 실시간으로 메시지를 주고받을 수 있으며, 파일 공유, 검색, 읽음 상태 확인, 알림 수신, 관리자 감사 기능 등을 이용할 수 있어야 한다.

본 시스템은 일반적인 메신저 기능을 넘어서, **조직도 기반 사용자 관리**, **부서 및 직책 기반 권한 제어**, **사내 보안 정책 반영**, **감사 로그 및 메시지 보존 정책**, **업무 협업 기능 확장 가능성**을 고려하여 설계되어야 한다.

---

# 2. 시스템 목표

1. 사내 구성원 간 빠르고 안정적인 실시간 커뮤니케이션 제공
2. 조직 구조를 반영한 사용자 및 채널 운영 지원
3. 사내 보안 정책에 부합하는 데이터 보호 및 접근 제어 제공
4. 메시지, 파일, 공지사항의 체계적 관리 및 검색 기능 제공
5. 향후 화상회의, 캘린더, 결재, 업무관리 시스템과 연동 가능한 구조 확보

# 5. 기능 요구사항

---

## 5.1 사용자 인증 및 계정 관리

### 5.1.1 로그인

* 사용자는 사내 계정으로 로그인할 수 있어야 한다.
* 로그인 방식은 사내 정책에 따라 다음 중 하나 또는 복수 지원 가능해야 한다.

  * 아이디/비밀번호 로그인
  * SSO 로그인
  * OAuth 기반 사내 인증 연동
* 로그인 실패 횟수가 일정 횟수를 초과하면 계정 잠금 또는 추가 인증이 요구되어야 한다.

### 5.1.2 세션 관리

* 로그인 후 사용자는 일정 시간 동안 세션을 유지할 수 있어야 한다.
* 일정 시간 이상 비활성 상태일 경우 자동 로그아웃 처리되어야 한다.
* 다중 기기 로그인 정책은 관리자 설정으로 제어 가능해야 한다.
* 비정상적인 세션 접근 탐지 시 강제 로그아웃이 가능해야 한다.

### 5.1.3 사용자 프로필

* 사용자는 자신의 프로필 정보를 조회할 수 있어야 한다.
* 표시 정보에는 다음이 포함될 수 있어야 한다.

  * 이름
  * 부서
  * 직책
  * 이메일
  * 프로필 이미지
  * 상태 메시지
  * 재직 상태
* 일부 항목은 사용자가 수정 가능하고, 일부는 조직도 시스템 연동값으로 읽기 전용이어야 한다.

---

## 5.2 조직도 및 사용자 관리

### 5.2.1 조직 정보 반영

* 시스템은 부서, 팀, 직급, 직책, 재직 상태 정보를 관리할 수 있어야 한다.
* 조직 정보는 외부 인사 시스템과 동기화 가능해야 한다.
* 부서 이동, 휴직, 퇴사, 입사 등의 상태 변화가 사용자 접근 권한에 반영되어야 한다.

### 5.2.2 사용자 검색

* 사용자는 이름, 이메일, 부서명, 직책 등을 기준으로 다른 사용자를 검색할 수 있어야 한다.
* 검색 결과에는 최소한 이름, 부서, 직책, 프로필 이미지가 보여야 한다.

### 5.2.3 사용자 상태 표시

* 사용자의 현재 상태를 표시할 수 있어야 한다.

  * 온라인
  * 자리 비움
  * 오프라인
  * 회의 중
  * 방해 금지
* 상태는 자동 변경 또는 수동 변경 가능해야 한다.

---

## 5.3 채팅방 기능

### 5.3.1 1:1 채팅

* 사용자는 다른 사용자와 1:1 대화를 시작할 수 있어야 한다.
* 동일 사용자 간 1:1 채팅방은 중복 생성되지 않고 기존 대화방으로 연결되는 것이 바람직하다.
* 차단 정책이 있는 경우, 차단된 사용자는 새로운 대화를 시작할 수 없어야 한다.

### 5.3.2 그룹 채팅

* 사용자는 다수의 사용자를 초대하여 그룹 채팅방을 만들 수 있어야 한다.
* 그룹 채팅방은 이름을 지정할 수 있어야 한다.
* 그룹 채팅방의 멤버 초대 및 내보내기 권한은 방장 또는 관리자 정책에 따라 달라져야 한다.

### 5.3.3 채널형 채팅방

* 부서 채널, 프로젝트 채널, 공지 채널 등 목적별 채널을 생성할 수 있어야 한다.
* 채널은 다음과 같은 속성을 가질 수 있어야 한다.

  * 공개 채널
  * 비공개 채널
  * 읽기 전용 채널
  * 공지 전용 채널
* 채널 참여 대상은 부서 또는 권한 그룹으로 제한할 수 있어야 한다.

### 5.3.4 채팅방 목록

* 사용자는 자신이 속한 채팅방 목록을 볼 수 있어야 한다.
* 목록에는 다음 정보가 표시되어야 한다.

  * 채팅방 이름
  * 최근 메시지
  * 최근 메시지 시간
  * 읽지 않은 메시지 수
  * 고정 여부
* 사용자는 자주 사용하는 채팅방을 상단에 고정할 수 있어야 한다.

---

## 5.4 메시지 기능

### 5.4.1 메시지 송수신

* 사용자는 텍스트 메시지를 실시간으로 송수신할 수 있어야 한다.
* 메시지 전송 후 상대방 또는 서버 수신 상태를 확인할 수 있어야 한다.
* 네트워크 지연이 있더라도 메시지 유실 없이 재전송 로직이 고려되어야 한다.

### 5.4.2 메시지 형식

* 메시지는 일반 텍스트를 기본으로 지원해야 한다.
* 추가적으로 다음 형식을 고려할 수 있어야 한다.

  * 줄바꿈
  * 링크 자동 인식
  * 이모지
  * 멘션(@사용자)
  * 해시태그
  * 코드 블록 또는 서식 텍스트 (선택)

### 5.4.3 메시지 수정

* 작성자는 자신이 보낸 메시지를 수정할 수 있어야 한다.
* 수정된 메시지는 수정 여부가 표시되어야 한다.
* 관리자 정책에 따라 수정 가능 시간 제한을 둘 수 있어야 한다.

### 5.4.4 메시지 삭제

* 작성자는 자신이 보낸 메시지를 삭제할 수 있어야 한다.
* 삭제 시 화면에는 “삭제된 메시지입니다”와 같은 표시가 가능해야 한다.
* 감사 정책상 실제 데이터는 즉시 물리 삭제하지 않고 소프트 삭제할 수 있어야 한다.
* 관리자는 정책에 따라 삭제된 메시지 원문을 별도 감사 화면에서 확인할 수 있어야 한다.

### 5.4.5 답장 및 전달

* 특정 메시지에 답장하는 기능이 있어야 한다.
* 기존 메시지를 다른 채팅방으로 전달할 수 있어야 한다.
* 전달 시 원본 작성자, 전달 시간, 첨부 포함 여부를 표시할 수 있어야 한다.

### 5.4.6 읽음 상태

* 시스템은 메시지별 또는 마지막 읽은 위치 기준으로 읽음 여부를 관리해야 한다.
* 1:1 채팅에서는 상대방이 읽었는지 표시할 수 있어야 한다.
* 그룹 채팅에서는 읽은 인원 수 또는 읽은 사용자 목록을 볼 수 있어야 한다.
* 읽음 처리는 사용자의 실제 대화방 진입 및 메시지 확인 시점에 반영되어야 한다.

### 5.4.7 멘션 및 강조

* 사용자는 채널 내에서 특정 사용자를 멘션할 수 있어야 한다.
* 멘션된 사용자는 일반 메시지보다 높은 우선순위의 알림을 받을 수 있어야 한다.
* 공지 메시지 또는 중요 메시지는 별도로 강조 표시되어야 한다.

---

## 5.5 파일 및 이미지 공유

### 5.5.1 파일 업로드

* 사용자는 채팅방에 파일을 업로드할 수 있어야 한다.
* 허용 파일 형식과 최대 크기는 관리자 정책으로 제어 가능해야 한다.
* 악성 파일 업로드 방지를 위한 검사 절차가 필요하다.

### 5.5.2 파일 다운로드

* 채팅방 참여자는 업로드된 파일을 다운로드할 수 있어야 한다.
* 권한이 없는 사용자는 파일에 접근할 수 없어야 한다.
* 다운로드 이력은 감사 목적상 로그로 남길 수 있어야 한다.

### 5.5.3 이미지 미리보기

* 이미지 파일은 채팅창 내 미리보기를 지원할 수 있어야 한다.
* 너무 큰 이미지는 썸네일을 생성하여 표시해야 한다.

### 5.5.4 파일 보존 정책

* 파일은 회사 정책에 따라 일정 기간 보관되어야 한다.
* 퇴사자 또는 접근 권한이 제거된 사용자는 기존 파일 접근 권한이 차단되어야 한다.

---

## 5.6 검색 기능

### 5.6.1 메시지 검색

* 사용자는 자신의 접근 권한이 있는 범위 내에서 메시지를 검색할 수 있어야 한다.
* 검색 조건에는 다음이 포함될 수 있어야 한다.

  * 키워드
  * 작성자
  * 채팅방
  * 날짜 범위
  * 파일 포함 여부

### 5.6.2 파일 검색

* 파일명 기준 검색이 가능해야 한다.
* 파일 업로드 사용자, 업로드 날짜, 채팅방 기준 필터링이 가능해야 한다.

### 5.6.3 검색 결과 이동

* 검색 결과를 선택하면 해당 메시지 위치로 이동할 수 있어야 한다.
* 검색어 하이라이트가 표시되어야 한다.

---

## 5.7 알림 기능

### 5.7.1 실시간 알림

* 새 메시지 수신 시 사용자는 실시간 알림을 받을 수 있어야 한다.
* 알림 방식은 다음을 지원할 수 있어야 한다.

  * 웹 푸시
  * 앱 푸시
  * 브라우저 알림
  * 내부 알림센터

### 5.7.2 알림 설정

* 사용자는 채팅방 단위로 알림 설정을 변경할 수 있어야 한다.

  * 전체 알림
  * 멘션만 알림
  * 무음
* 사용자는 근무시간 외 알림 제한 정책을 설정할 수 있어야 한다.

### 5.7.3 중요 알림

* 공지 채널 또는 관리자 메시지는 일반 알림과 구분되어야 한다.
* 긴급 공지는 읽음 확인을 강제하는 옵션이 있을 수 있어야 한다.

---

## 5.8 공지 및 고정 기능

### 5.8.1 공지 메시지

* 관리자 또는 권한 있는 사용자는 특정 메시지를 공지로 지정할 수 있어야 한다.
* 공지 메시지는 채팅방 상단 또는 별도 공지 영역에 노출되어야 한다.

### 5.8.2 메시지 고정

* 채팅방 내 중요한 메시지는 고정할 수 있어야 한다.
* 사용자는 고정된 메시지 목록을 별도 화면에서 확인할 수 있어야 한다.

### 5.8.3 읽음 확인 공지

* 특정 공지는 구성원이 읽었는지 확인 가능해야 한다.
* 관리자는 미확인 사용자 목록을 조회할 수 있어야 한다.

---

## 5.9 관리자 기능

### 5.9.1 사용자 관리

* 관리자는 사용자 계정을 생성, 비활성화, 잠금 해제할 수 있어야 한다.
* 관리자는 사용자의 부서/권한/상태를 조회하고 조정할 수 있어야 한다.

### 5.9.2 채널 관리

* 관리자는 공개 채널 및 공지 채널을 생성/삭제할 수 있어야 한다.
* 관리자는 채널 참여 권한, 글쓰기 권한, 파일 업로드 권한을 제어할 수 있어야 한다.

### 5.9.3 정책 관리

* 관리자는 다음 정책을 관리할 수 있어야 한다.

  * 메시지 보존 기간
  * 파일 보존 기간
  * 업로드 용량 제한
  * 허용 파일 형식
  * 비밀번호 정책
  * 세션 만료 정책
  * 접속 허용 IP 정책
  * 외부 다운로드 제한 정책

### 5.9.4 모니터링

* 관리자는 현재 접속자 수, 메시지 처리량, 오류 발생 현황 등을 확인할 수 있어야 한다.
* 장애 발생 시 관리자에게 알림이 전송되어야 한다.

---

## 5.10 감사 및 로그 기능

### 5.10.1 감사 로그

* 시스템은 주요 행위를 감사 로그로 기록해야 한다.

  * 로그인/로그아웃
  * 메시지 작성/수정/삭제
  * 파일 업로드/다운로드
  * 채널 생성/삭제
  * 권한 변경
  * 관리자 정책 변경

### 5.10.2 감사 조회

* 감사 담당자는 기간, 사용자, 채널, 행위 유형 기준으로 로그를 조회할 수 있어야 한다.
* 감사 로그는 위변조 방지를 고려하여 저장되어야 한다.

### 5.10.3 메시지 보존

* 법적 또는 사내 정책에 따라 삭제된 메시지라도 일정 기간 보존할 수 있어야 한다.
* 보존 정책 만료 시 안전하게 파기할 수 있어야 한다.

# 6. 비기능 요구사항

---

## 6.1 성능

* 일반 메시지 전송은 평균 1초 이내에 상대방 화면에 반영되는 것을 목표로 한다.
* 동시 접속자 수 증가에 따라 수평 확장이 가능해야 한다.
* 검색 응답은 일반 조건 기준 3초 이내를 목표로 한다.

## 6.2 가용성

* 서비스는 업무시간 중 안정적으로 동작해야 하며 고가용성을 고려해야 한다.
* 장애 발생 시 자동 복구 또는 신속한 장애 전환이 가능해야 한다.

## 6.3 확장성

* 사용자 수, 채널 수, 메시지량 증가에 따라 확장 가능해야 한다.
* 메시지 서버, 파일 서버, 검색 서버를 분리 가능한 구조가 바람직하다.

## 6.4 보안

* 모든 통신은 암호화되어야 한다.
* 비밀번호는 단방향 암호화 저장해야 한다.
* 권한 없는 사용자는 다른 부서의 비공개 채널에 접근할 수 없어야 한다.
* 파일 접근은 인증 및 권한 검증 후 허용되어야 한다.
* XSS, CSRF, 파일 업로드 취약점, 세션 탈취 등에 대한 대응이 필요하다.

## 6.5 사용성

* 사용자는 별도 교육 없이 기본 채팅 기능을 사용할 수 있어야 한다.
* UI는 직관적이어야 하며 데스크톱 환경에서 업무용으로 적합해야 한다.
* 자주 사용하는 기능은 최소 클릭으로 접근 가능해야 한다.

## 6.6 호환성

* 최신 브라우저 환경에서 동작해야 한다.
* 데스크톱 웹을 우선 지원하되, 향후 모바일 앱 확장을 고려해야 한다.

## 6.7 유지보수성

* 기능별 모듈 분리가 명확해야 한다.
* 로그와 모니터링 체계를 통해 장애 원인 추적이 쉬워야 한다.
* 설정값 변경이 코드 수정 없이 가능해야 한다.

# 12. 개발 관점에서의 핵심 포인트

이 요구사항을 실제로 구현하려면 보통 아래가 핵심이 된다.

* **실시간 처리**: WebSocket 또는 SSE 기반 메시지 전달

* **조직도 기반 권한**: RBAC 또는 부서 기반 접근 제어
* **메시지 저장 구조**: 채팅방, 멤버, 메시지, 읽음 상태 분리
* **파일 관리**: 업로드 서버 또는 오브젝트 스토리지
* **검색**: DB 검색 + 전문 검색 엔진 고려 가능
* **감사 로그**: 별도 저장소 또는 이벤트 로그 설계
* **확장성**: 채팅 서버와 일반 API 서버의 역할 분리
