# 04. 세션 내 대화 이력 추가

## 이번 단계의 목표

기존 REPL은 반복적으로 질문하고 응답받을 수는 있었지만, 각 줄을 독립된 요청으로 처리했다.  
즉 사용자는 대화를 이어가는 것처럼 느끼지만, 모델은 이전 대화를 기억하지 못했다.

이번 단계의 목표는 이 흐름을 한 단계 확장해서, **같은 REPL 세션 안에서 이전 user/assistant 메시지를 메모리에 쌓고 다음 호출에 함께 전달하는 것**이다.

즉 이번 단계의 핵심은 다음 한 줄로 정리할 수 있다.

**REPL을 단순한 반복 입력 루프에서, 문맥이 이어지는 대화형 루프로 바꾼다.**

## 왜 이 단계가 필요한가

REPL이 생기면 사용자는 자연스럽게 이제 대화가 이어지겠지라고 기대한다.  
하지만 이전 단계까지는 그렇지 않았다. 프로그램은 계속 켜져 있어도, 각 요청은 사실상 처음부터 새로 시작했다.

대화 이력이 필요한 이유:

- 두 번째 질문부터는 이전 맥락을 이어서 묻고 답할 수 있다.
- REPL이 진짜 대화형 인터페이스처럼 동작하기 시작한다.
- 이후 도구 호출, 계획, 검증 같은 더 복잡한 흐름도 세션 문맥 위에서 쌓을 수 있다.
- 하네스의 상태 계층을 처음으로 실제 코드에 도입하게 된다.

즉 이번 단계는 반복형 CLI를 문맥이 유지되는 하네스로 바꾸는 첫 번째 상태 관리 단계다.

## 이번 단계의 범위

### 포함

- REPL 세션 동안 user/assistant 메시지를 메모리에 유지
- 다음 호출 때 이전 이력을 Anthropic에 함께 전달
- 단일 프롬프트 모드도 내부적으로 메시지 배열 구조 사용
- 이력 전달 동작을 검증하는 자동 테스트 추가

### 제외

- 프로그램 종료 후에도 대화를 저장하는 기능
- 세션 파일 저장/불러오기
- 도구 호출
- 권한/정책
- 멀티 에이전트

## 현재 구조와 이번 단계의 변화

기존 REPL 구조:

```text
한 줄 입력
-> 그 줄만 Anthropic에 전달
-> 응답 출력
-> 다음 줄 입력
```

변경 후 REPL 구조:

```text
한 줄 입력
-> 이전 세션 이력 + 새 user 메시지 결합
-> Anthropic에 전체 메시지 배열 전달
-> 응답 출력
-> user/assistant 쌍을 세션 이력에 저장
-> 다음 줄 입력
```

핵심 차이는 모델 호출 단위가 문자열 프롬프트 하나에서 메시지 배열 전체로 바뀌었다는 점이다.

## 핵심 구현 내용

### 1. 메시지 타입 분리

`src/lib/chat-message.ts`를 새로 추가했다.

이 파일은 다음 타입을 정의한다.

```ts
type ChatMessage = {
  role: "user" | "assistant";
  content: string;
};
```

이 타입을 중심으로 이제 하네스 내부에서는 하나의 프롬프트 문자열보다 메시지 목록을 기준으로 작업한다.

### 2. Anthropic 호출기 인터페이스 변경

`src/lib/anthropic.ts`의 함수들은 이제 문자열 대신 메시지 배열을 받는다.

- `generateText(messages, config)`
- `streamText(messages, config, onChunk)`

특히 `streamText()`는 이번 단계에서 **최종 assistant 텍스트를 반환**하도록 바뀌었다.  
이유는 REPL이 스트리밍 출력만 하는 것이 아니라, 완료된 assistant 응답을 세션 이력에 저장해야 하기 때문이다.

즉 이번 단계에서 모델 호출 계층은 다음 두 가지 역할을 동시에 맡는다.

- 청크 단위 스트리밍 출력
- 완료된 assistant 텍스트 반환

### 3. 단일 프롬프트 모드도 메시지 배열 기준으로 통일

`src/lib/run-prompt-command.ts`는 이제 단일 문자열 프롬프트를 바로 넘기지 않고, 내부에서 아래처럼 변환한다.

```ts
[{ role: "user", content: prompt }]
```

이렇게 해서 단일 프롬프트 모드와 REPL 모드가 같은 메시지 구조를 사용하게 됐다.

### 4. REPL 세션 이력 유지

`src/lib/run-repl-command.ts`는 세션 메모리로 `history` 배열을 유지한다.

동작 방식:

1. 사용자가 새 줄을 입력한다.
2. 기존 `history` 뒤에 새 user 메시지를 붙여 이번 호출용 메시지 배열을 만든다.
3. Anthropic 스트리밍 호출을 실행한다.
4. 완료된 assistant 텍스트를 받는다.
5. 이번 user/assistant 쌍을 `history`에 추가한다.

즉 REPL이 살아 있는 동안만, 그 세션 내부의 대화를 계속 기억한다.

### 5. 자동 테스트 확장

`tests/repl-cli.test.ts`에 다음 검증을 추가했다.

- 두 번째 호출 시 첫 번째 user/assistant 메시지가 함께 전달되는가

이 테스트가 이번 단계의 핵심이다.  
즉 단순히 REPL이 돈다는 것만이 아니라, 실제로 문맥이 메시지 배열에 반영되는지를 코드로 고정했다.

## 동작 규칙

이번 단계 기준 REPL 규칙은 다음과 같다.

- REPL이 살아 있는 동안만 대화 이력을 유지한다.
- 프로그램을 종료하면 이력은 사라진다.
- 단일 프롬프트 모드는 여전히 단발성이다.
- REPL의 각 assistant 응답은 다음 호출의 문맥으로 포함된다.

즉 이번 단계는 세션 내 기억만 추가한 것이고, 세션 간 저장은 아직 아니다.

npm test

## 완료 기준

아래 조건을 모두 만족하면 이번 단계는 완료다.

- REPL 세션 동안 이전 user/assistant 메시지를 유지한다.
- 다음 호출에 이전 이력을 함께 전달한다.
- 단일 프롬프트 모드도 메시지 배열 구조를 사용한다.
- 테스트로 이력 전달이 검증된다.

현재 상태에서는 위 조건을 모두 충족했다.

## 다음 단계 후보

이번 단계 다음으로 자연스러운 후보는 아래와 같다.

1. 세션 간 저장 추가
2. 도구 호출 기초 추가

특히 세션 간 저장은 지금 만든 세션 내 기억을 프로그램 재시작 이후까지 확장하는 단계다.

## 메모

이번 단계의 대화 이력은 오직 메모리 안에만 존재한다.  
즉 REPL을 종료하면 문맥은 사라지고, 다시 시작하면 빈 세션부터 시작한다.

따라서 다음 단계에서 물어볼 핵심 질문은 이것이다.

**이제 기억은 생겼으니, 그 기억을 종료 후에도 어떻게 보존할 것인가?**
